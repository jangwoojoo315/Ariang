# 자동 커밋 생성

스테이징된 변경사항을 분석해서 커밋 메시지를 작성하고 커밋을 생성하라.

---

## 준비 단계

다음 명령을 실행해서 정보를 수집하라:

```bash
git diff --staged
```

```bash
git diff --staged --stat
```

수집한 정보와 이번 대화에서 사용자가 요청한 내용을 함께 분석한다.

---

## 커밋 메시지 형식 판단

### 버그 수정 / 문제 해결인 경우

```
fix: <한줄 요약>

[이슈]
<어떤 문제가 있었는지 구체적으로>

[수정]
<어떤 방식으로 수정했는지>

[유의사항]
<잠재적 사이드 이펙트, 추가 수정이 필요한 사항, 주의할 점>
```

- [유의사항]에 특별히 언급할 게 없으면 해당 섹션 생략

### 신규 기능 추가인 경우

```
feat: <한줄 요약>

<상세 설명 — 최대 3줄, 없으면 생략>
```

### 기타 접두어 기준

| 변경 유형 | 접두어 |
|-----------|--------|
| 리팩토링 (동작 변경 없음) | `refactor:` |
| 스타일 / 포맷 | `style:` |
| 설정, 빌드, 의존성 | `chore:` |
| 문서 | `docs:` |

---

## 실행 규칙

1. 스테이징된 파일이 없으면 "스테이징된 변경사항이 없습니다. `git add` 후 다시 실행하세요." 안내 후 중단
2. 작성한 커밋 메시지를 사용자에게 먼저 보여주고 확인을 받는다
3. 승인 후 커밋 생성:

```bash
git commit -m "$(cat <<'EOF'
<작성한 커밋 메시지 전체>
EOF
)"
```

4. 커밋 성공 후 `git log --oneline -1` 로 결과 확인

---

## 작성 원칙

- 커밋 메시지는 한국어로 작성
- 한줄 요약은 50자 이내
- 파일명·함수명 나열보다 **변경의 이유와 영향** 중심으로 작성
- 대화 맥락이 없으면 diff만으로 판단

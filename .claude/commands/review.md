# 프로젝트 리뷰

아래 3단계를 순서대로 실행하고, 결과를 **필수 / 권장 / 선택** 3가지 카테고리로 정리해서 보고하라.

---

## 1단계 — 타입 및 린트 검사

다음 두 명령을 순서대로 실행하라:

```bash
npx tsc --noEmit --project tsconfig.json
```

```bash
pnpm lint
```

각 명령의 출력(에러, 경고 전체)을 수집한다. 성공 시 "통과"로 기록한다.

---

## 2단계 — FSD 구조 검사

`src/` 디렉토리를 탐색해서 아래 규칙 위반 여부를 확인하라:

**레이어 위치 규칙**
- `app/` — Next.js 라우트 파일(`page.tsx`, `layout.tsx`, `loading.tsx`, `error.tsx`, `not-found.tsx`, `route.ts`)과 전역 설정만 허용. 실질적인 UI 컴포넌트 로직이 들어 있으면 위반.
- `views/` — 페이지 단위 조합 컴포넌트. 직접 API 호출하거나 비즈니스 로직을 포함하면 위반.
- `widgets/` — 복합 UI 블록. `views/`를 import하면 위반.
- `features/` — 사용자 액션/시나리오. `widgets/` 또는 `views/`를 import하면 위반.
- `entities/` — 도메인 모델. `features/` 이상을 import하면 위반.
- `shared/` — 범용 코드. 어떤 상위 레이어도 import하면 위반.

**단방향 의존성 확인** (각 파일의 import 경로를 검사)
- `@/views/*` import는 `app/` 에서만 허용
- `@/widgets/*` import는 `app/`, `views/` 에서만 허용
- `@/features/*` import는 `app/`, `views/`, `widgets/` 에서만 허용
- `@/entities/*` import는 `app/`, `views/`, `widgets/`, `features/` 에서만 허용
- `@/shared/*` import는 모든 레이어에서 허용

**barrel 파일 규칙**
- 세그먼트 폴더(`ui/`, `model/`, `api/`, `lib/`, `config/`, `types/`) 내부에 `index.ts`가 있으면 위반
- 슬라이스 루트에 `index.ts`가 없으면 경고

**`pages/` 폴더 존재 여부**
- `src/pages/` 폴더가 있으면 위반 (Next.js 예약어이므로 `views/`를 사용해야 함)

---

## 3단계 — 코드 리뷰

변경된 파일 또는 전체 `src/` 코드를 읽고 아래 관점에서 검토하라:

- 타입 안전성 (any 남용, 불필요한 타입 단언)
- React 패턴 (불필요한 리렌더, 잘못된 훅 사용, key 관리)
- 비동기 처리 (에러 핸들링 누락, 경쟁 조건)
- 접근성 (시맨틱 HTML, ARIA 속성)
- 성능 (불필요한 연산, 메모이제이션 과용)
- 보안 (XSS, 민감 정보 노출)

### 🚨 필수 (반드시 수정)

서비스 동작 오류, 빌드 실패, 보안 취약점, FSD 단방향 의존성 위반에 해당하는 항목.
수정하지 않으면 기능이 깨지거나 배포가 불가능한 문제.

- **[파일경로:라인]** 문제 설명 → 수정 방법

---

### 💡 권장 (수정하면 좋음)

코드 품질, 유지보수성, 성능, FSD 구조 경고에 해당하는 항목.
지금 당장 문제는 없지만 방치하면 기술 부채가 되는 것들.

- **[파일경로:라인]** 문제 설명 → 수정 방법

---

### 🔧 선택 (취향/상황에 따라)

일관성, 가독성, 스타일에 해당하는 항목.
수정해도 되고 안 해도 되지만 코드베이스 전반에 통일하면 좋은 것들.

- **[파일경로:라인]** 문제 설명 → 수정 방법
